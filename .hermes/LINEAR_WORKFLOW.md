## 3. Issue Naming Convention

* Format: `<area/module> — <short description>`
* Examples:
  * `pipeline/downloader.py — yt-dlp Download Module`
  * `Configuration Management System`
  * `Portrait Smart Crop & SPLIT Mode`
* Keep titles short (< 60 chars where possible).
* Issues are module-level, not task-level. A single issue should map to a
  module or a coherent chunk of work — never a single checklist item.

## 5. Status Rules

The team has these statuses: Backlog, Todo, In Progress, In Review, Done,
Canceled, Duplicate.

Recommended movement:

* `Backlog` → `Todo`: when an issue is ready to be picked up next.
* `Todo` → `In Progress`: when an agent or developer starts work. **Only
  one issue per agent/developer should be In Progress at a time.**
* `In Progress` → `In Review`: when code is written and tests pass; ready
  for review or integration validation.
* `In Review` → `Done`: when the issue is closed out and acceptance
  criteria are met.
* `*` → `Canceled` / `Duplicate`: when abandoned. Always leave a comment
  explaining why.

Hard rules:

* Never move an issue backwards from `Done` to `In Progress` without a
  new comment explaining the reason.
* If an issue is `Blocked`, set status to `Todo` but also add a comment
  describing the blocker. Only use the `Done` status when the work is
  actually verified and accepted.

```
**What I did:**
- <bullet>

**Decisions made:**
- <ADR-XXX reference or new decision>

**Tests run:**
- <command> — <result>

**Blockers:**
- <description or "none">

**Next session should start with:**
- <first task>
```

For multi-issue sessions, post one comment per issue. The Linear activity
log is the project journal — keep it readable.

---

## 7. Rules for Moving Issues Between Statuses

* Always post a session comment **before** changing status, so the audit
  trail explains the transition.
* When moving to `In Review`, include a checklist of acceptance criteria
  and their current state (✅ / ❌).
* When moving to `Done`, link the commit / PR in the comment.
* When moving to `Canceled`, the comment must explain the reasoning
  and reference any superseding issue.
* An issue CANNOT move to `Done` while any of its `Depends on` issues is
  not yet `Done`. Verify with the dependency graph first.
